# 新闻来源与选题系统重构计划

## 文档目的

把 Daily News 从“抓取若干文章后按分数排列”的原型，重构为“按事件组织、覆盖每日重要议题、可解释地筛选信息”的一站式新闻简报。

目标不是收录尽可能多的链接，而是让用户在浏览本网站后，能够理解当天绝大多数真正重要的事件、其影响、关键事实和后续关注点；外部新闻网站主要用于核验原文或继续深读，而不是补齐本网站遗漏的基本信息。

本文最初作为实施前计划创建，现同时记录落地状态。核心事件管线、V2 API/UI、last-known-good、覆盖调度和可靠性门槛已实现；长期运营验证与外部持久化仍按本文继续推进。

## 适用范围

本计划覆盖：

- 新闻来源分层、准入、调度和健康度；
- 抓取、标准化、发布时间和来源归因；
- 同一事件聚类、多来源证据合并和事件持续更新；
- 重要性判断、内容质量门槛、覆盖约束和多样性重排；
- 中文摘要、事实约束、原文归因和失败降级；
- 日报结构、API 兼容、持久状态、可观测性和验证方法；
- 从现有管线迁移到新管线的分阶段路线。

本计划不直接决定最终存储供应商、最终日报条数、是否增加新栏目、翻译模型供应商和生产发布窗口；这些内容列为 `待确认`。

## Plan 和项目证据

### 用户目标

- 聚合当天最重要、最有价值的新闻，而不是事无巨细地收录所有文章。
- 不能因为来源或分类调度失衡，只剩少量零散新闻。
- 用户阅读本网站后，应能掌握绝大多数重要信息，不必再依赖多个新闻网站补齐每日认知。

### 当前仓库基线

| 证据 | 当前事实 | 对重构的影响 |
| --- | --- | --- |
| `src/config/sources.ts` | 启用 49 个来源、58 个分区，但 `policy` 和 `society` 没有独立主分区，AI 分区明显更多 | 不能再用“来源数量”代替“议题覆盖” |
| `scripts/newsApi.ts` | Serverless 默认只选择 6 个来源 | 固定来源切片会造成线上覆盖偏斜 |
| `scripts/newsService.ts` | 抓取以来源和分区为循环单位，Firecrawl/直连/fallback 为全局降级链 | 需要改成按 coverage beat 调度，并保留分阶段结果 |
| `src/lib/dedupe.ts` | 使用 URL 和 token overlap 做文章级聚类 | 对改写标题、跨语言和持续更新的同一事件识别不足 |
| `src/lib/scoring.ts` | 重要性主要来自分类基础分、关键词和多来源加分 | 不能可靠表达影响范围、后果持续时间、紧迫性和系统性意义 |
| `src/lib/newsPipeline.ts` | 聚类后直接评分、可信度过滤并输出一个扁平 `items` 列表 | 缺少事件层、证据层、编辑层级和覆盖约束 |
| `src/App.tsx` | 首页展示 5 条热点，其余按偏好或分类浏览 | 用户缺少“今日必知—重要进展—持续关注”的信息层级 |
| `public/daily-news.json` | 当前 66 条、13 个来源；国际 15、财经 13、体育 12，科学和娱乐为 0；4 条为通用模板摘要 | 当前数量不算少，但分布和信息密度不满足一站式目标 |
| 本地运行审查 | 全来源首轮刷新超过两分钟仍未形成可用缓存 | 抓取不能继续发生在用户请求的关键路径上 |

### 外部原则参考

| 参考 | 本计划采用的原则 |
| --- | --- |
| [Google News 官方排序说明](https://support.google.com/news/publisher-center/answer/9606702?hl=en) | 重要新闻选择同时考虑相关性、显著性、权威性、新鲜度和可用性，并需要来源多样性 |
| [Reuters Trust Principles](https://reutersagency.com/about/our-trust-principles/) | 系统设计要保护来源独立性、完整性和免受单一立场控制，不能让单一媒体决定事件事实 |
| [AP News Values and Principles](https://www.ap.org/about/news-values-and-principles/) | 互联网信息必须核验并归因到原始来源；准确性、公平性和明确纠错优先于速度和数量 |

这些原则用于定义产品质量，不代表引入 Google、Reuters 或 AP 的付费内容或 API。

## 产品北极星

将“一站式新闻”定义为可验收的产品结果：

> 用户在 10 到 15 分钟内读完首页后，能理解当天至少 90% 的“必须知道”事件；每个事件都能回答发生了什么、为什么重要、有哪些关键事实、信息有多可靠、接下来关注什么。

“至少 90%”不能通过来源数量直接计算。实施时需要建立每日人工标注的基准事件集，用独立来源组合形成 ground truth，再测量本系统是否覆盖。

## 目标信息结构

建议将首页从单一新闻流改为四层，数字为初始建议值，实施前可调整：

| 层级 | 建议数量 | 准入标准 | 用户价值 |
| --- | ---: | --- | --- |
| 今日必知 | 8–12 个事件 | 高影响、高可信、跨领域均应知道 | 读完即可掌握当天核心变化 |
| 重要进展 | 20–35 个事件 | 对特定行业、地区或公共议题有明显影响 | 补齐主要领域的重要信息 |
| 持续关注 | 5–10 个事件 | 潜在影响高，但事实仍在发展或证据不足 | 显示不确定性和下一观察点，不混入已确认事实 |
| 分类深读 | 每类只收录达标事件 | 通过质量门槛但未进入前两层 | 为需要更多领域细节的用户提供深度 |

同一事件只能在首页拥有一个主卡片。分类页可以引用该事件，但不能复制成多个独立新闻项。

### 事件卡片最低信息量

每张卡片至少包含：

1. 一句话说明发生了什么；
2. 一句话说明为什么重要；
3. 2–4 个可追溯的关键事实；
4. 影响对象或影响范围；
5. 事件状态：已确认、发展中、已更正；
6. 最新更新时间和最初发生时间；
7. 主要来源、交叉验证来源和原文链接；
8. 下一步值得关注的时间点或问题。

模型生成的文字不能引入来源中不存在的事实。无法满足事实约束时，应使用抽取式摘要或直接缩短卡片，而不是生成通用或推测性内容。

## 分类体系重构

当前分类把地理范围和主题混在同一维度，例如一条中国政策新闻需要在 `china` 和 `policy` 之间二选一。新模型应拆成四个独立轴：

| 轴 | 示例 | 用途 |
| --- | --- | --- |
| `primaryBeat` | 公共事务、经济金融、商业产业、科技、AI、科学健康、社会、气候灾害、文化、体育 | 决定唯一主栏目 |
| `scope` | 中国、国际、全球、地区性 | 表达影响范围，不参与用户偏好加减分 |
| `eventType` | 政策、冲突、灾害、财报、产品发布、研究、调查、赛事 | 支持重要性规则和摘要模板 |
| `topics/entities` | 公司、机构、人物、国家、技术、行业 | 聚类、搜索、持续跟踪和去重 |

地理和语言只用于覆盖审计、抓取和展示处理，不作为新闻价值或用户偏好权重。最终栏目数量和是否把 AI 保持为独立一级栏目：`待确认`。

## 来源体系重构

### 来源角色

来源不再只用统一 `credibility` 数字表示，而按在证据链中的角色管理：

| 角色 | 用途 | 能否单独进入“今日必知” |
| --- | --- | --- |
| 原始/官方来源 | 法规、公告、财报、研究论文、机构声明、赛事结果 | 可以证明“已发布/已发生”，但涉及争议和影响判断时仍需独立来源 |
| 核心综合来源 | 跨领域发现、快速确认、重大事件连续报道 | 高质量报道可以；重大争议事件优先要求交叉验证 |
| 垂直专业来源 | AI、科技、财经、科学、体育等领域深度 | 可以进入对应领域，跨领域重大性需额外证据 |
| 地区一线来源 | 补充本地事实和现场信息 | 需要来源健康记录与必要的交叉验证 |
| 发现型来源 | 搜索、聚合页、社交媒体、人物动态 | 只产生线索，不能作为已确认事件的唯一证据 |

### 来源配置字段

建议将 `NewsSource` 扩展为以下逻辑字段，具体类型在实现阶段确定：

- `roles`：来源承担的证据角色；
- `beats`：负责覆盖的领域；
- `scopeCoverage`：地理覆盖能力；
- `originality`：原始报道/转载/聚合倾向；
- `authorityPrior`：历史权威度先验；
- `accessReliability`：最近抓取成功率和响应时间；
- `verificationPolicy`：何时允许单源、何时要求双源或官方源；
- `discoveryMethods`：RSS、栏目页、search、官方 API 等；
- `language` 和 `translationRequired`：仅用于处理链，不用于价值排序；
- `enabledReason`、`disabledReason`、`lastReviewedAt`：来源治理和审计。

### 覆盖矩阵和调度

调度器以 `beat × scope × source role` 为基本单元，而不是先取固定前 N 个来源：

1. 每个 beat 至少配置两个独立发现入口和一个可验证入口；
2. 每轮根据来源健康度、上次成功时间和覆盖缺口选择任务；
3. 重大突发 beat 可临时提高刷新频率，但不能挤占所有其他领域；
4. 单个来源失败只影响对应 coverage slot；
5. 如果某领域没有达标事件，允许为空，但必须记录“无合格候选”，不能用低价值新闻填数。

## 目标处理管线

```mermaid
flowchart LR
  A["来源注册表与覆盖矩阵"] --> B["健康感知调度器"]
  B --> C["候选抓取与标准化"]
  C --> D["硬质量门槛"]
  D --> E["事件聚类与持续更新"]
  E --> F["证据合并与事实状态"]
  F --> G["重要性分级"]
  G --> H["覆盖与多样性重排"]
  H --> I["事实约束摘要"]
  I --> J["日报发布与质量报告"]
```

### 1. 候选抓取与标准化

- 并发池抓取，设置单任务超时、整轮截止时间和来源熔断；
- 统一 canonical URL、标题、发布时间、作者/机构、正文摘要和来源域名；
- 保留 `discoveredAt`、`publishedAt`、`updatedAt`，不再把抓取时间当发布时间；
- 校验结果域名与来源配置，防止错误归因；
- 把原始抓取结果和可展示内容分开，避免清洗过程丢失证据。

### 2. 硬质量门槛

以下候选不能进入正式事件：

- 无法确认来源域名、标题、URL 或发布时间；
- 超出新鲜度窗口且不是持续事件的重要更新；
- 纯营销软文、导航页、列表页、重复稿或只有标题没有事实；
- 社交媒体单点爆料且没有后续确认；
- 需要绕过登录、付费墙或访客验证才能获得关键事实；
- 中文化失败且无法提供可靠中文摘要。

### 3. 事件聚类

聚类对象从“相似文章”升级为“同一现实事件”：

- 先按 canonical URL、实体、时间窗口和事件类型生成候选集合；
- 再使用标题/摘要语义相似度判断同一事件；
- 区分“同一事件的更新”和“同一主题的不同事件”；
- 事件保留多个 `evidence`，但只产生一个主卡片；
- 新进展更新已有事件，而不是每天重新生成一个重复卡片；
- 聚类错误必须可以通过 fixture 重现和人工纠正。

### 4. 证据和事实状态

每个事件需要形成可审计证据链：

- `confirmed`：由官方/原始来源或至少两个独立可靠来源确认；
- `developing`：影响可能较大，但关键事实仍在变化；
- `disputed`：可靠来源对关键事实存在冲突；
- `corrected`：已发生可见更正；
- `unverified`：只能进入内部候选或“持续关注”，不能进入“今日必知”。

摘要中的关键事实要能映射到具体 evidence URL。不同来源仅重复同一通稿时，不计作独立确认。

## 重要性判断

### 先分级，再排序

避免把所有事件压成一个看似精确但不可解释的总分。先判断事件属于哪个层级，再在层级内排序：

| 层级 | 定义 |
| --- | --- |
| `must_know` | 对大量人群、关键制度、公共安全、经济系统或国际局势有显著影响，即使用户没有偏好也应看到 |
| `important` | 对某一主要行业、地区或公共议题有明显影响，值得大多数关注该领域的用户阅读 |
| `special_interest` | 对特定专业或兴趣人群有价值，但不应挤占全站核心位置 |
| `noise` | 重复、低影响、纯宣传、缺少事实或仅有微小增量，不进入正式日报 |

### 重要性特征

初始实现至少评估：

- 影响人数和地理范围；
- 对政策、公共安全、经济、行业或基础设施的影响强度；
- 后果持续时间和不可逆性；
- 是否改变已有规则、预期或趋势；
- 紧迫性和用户是否需要采取行动；
- 是否为首次发生、重大突破或关键进展；
- 多来源显著性与独立确认；
- 信息不确定性、争议和来源风险。

用户偏好只能影响 `important` 和 `special_interest` 的排序，不能隐藏 `must_know`。来源可信度是事实准入和置信度维度，不与新闻重要性混为同一个分数。

### 选择约束

完成分级后再应用集合级约束：

- 同一事件只出现一次；
- 同一实体、同一来源和同一 beat 设置软上限；
- 不为满足分类数量而加入 `noise`；
- 某一大型事件可以有“主事件 + 一个确有独立价值的分析角度”，但必须显式建立关系；
- 当多条事件价值接近时，优先补齐覆盖缺口和独立视角；
- “今日必知”不受个人偏好影响，“重要进展”可做轻量个性化。

## 摘要和中文化策略

推荐把当前“直接让模型翻译/补摘要”改成事实约束流程：

1. 从 evidence 提取结构化事实、时间、实体、数字和出处；
2. 去除来源间重复或冲突的陈述；
3. 仅允许模型使用已提取事实生成中文表达；
4. 对人名、数字、日期、机构和关键结论做自动一致性检查；
5. 校验失败时回退到抽取式摘要；
6. 保留来源归因，不把二手报道包装成本站原创事实。

模型可以帮助翻译、实体抽取、事件类型识别和摘要压缩，但不能独自决定事实是否成立、来源是否可靠或事件是否属于 `must_know`。

## 运行架构

抓取和选题必须与用户请求解耦：

```text
定时/手动采集任务
  -> 写入候选、事件、证据和来源健康状态
  -> 生成不可变 DailyBrief snapshot
  -> 原子切换 latest report

GET /api/news
  -> 只读取 latest report
  -> 不在请求内抓取 49 个来源
```

必要行为：

- API 在冷启动时也能立即返回上次成功报告；
- 新报告只有通过数据质量门槛后才能替换旧报告；
- 刷新失败保留旧报告并暴露非敏感的运行状态；
- 抓取任务有并发上限、30 秒级整轮预算、来源熔断和重试退避；
- 本地文件/内存实现与生产持久存储通过同一 `NewsStore` 接口隔离；
- 生产持久化供应商和任务调度方式：`待确认`。

逻辑数据模型见 [database-design.md](database-design.md)。

## API 和兼容策略

目标报告从扁平 `items` 升级为分层结构：

```ts
interface DailyNewsReportV2 {
  version: 2;
  generatedAt: string;
  window: { from: string; to: string };
  stories: StoryCard[];
  topStories: StoryCard[];
  importantStories: StoryCard[];
  watchlist: StoryCard[];
  sections: Array<{ beat: string; storyIds: string[] }>;
  coverage: CoverageSummary;
  quality: PublicQualitySummary;
  items: RankedNewsItem[]; // 迁移期兼容字段
}
```

`stories` 是所有正式事件的规范集合，`sections.storyIds` 引用它；迁移期保留 `items` 兼容字段。详细当前契约见 [api-design.md](api-design.md)。

`coverage` 只暴露分类覆盖、报告新鲜度和来源数量等安全信息；内部错误、抓取 URL 细节、secret 和模型请求不得进入公开响应。

## 可观测性和质量门槛

每次选题运行至少记录：

- 每个来源/beat 的任务数、成功数、超时数和熔断状态；
- 候选数以及因域名、时间、语言、质量、重复、置信度被过滤的数量；
- 聚类前文章数、事件数、平均证据数和疑似错误聚类数；
- 各重要性层级、beat、scope 和来源的分布；
- 生成摘要的校验通过率与抽取式 fallback 比例；
- 新报告与上一版的新增、更新、删除和异常波动。

建议初始发布门槛：

| 指标 | 建议阈值 |
| --- | ---: |
| `must_know` 基准事件召回率 | ≥ 90% |
| 首页人工价值精确率 | ≥ 85% |
| 重复事件率 | ≤ 5% |
| 错误来源归因率 | 0% |
| 缺失发布时间 | 0% |
| 通用模板摘要 | ≤ 3% |
| 单一来源占全部正式事件比例 | ≤ 20% |
| 有新鲜合格候选的 beat 覆盖率 | 100% |
| API 读取 latest report 的 P95 | ≤ 500 ms |
| 单轮采集/生成 P95 | ≤ 30 秒，具体值待真实基准校准 |

这些阈值是推荐起点，需要通过人工标注的 7–14 天 golden dataset 校准，不能为了通过数字而加入低价值内容。

## 实施阶段

### Phase 0：建立基准和防回退测试（部分完成）

工作：

- 保存 7–14 天代表性候选 fixture；
- 人工标注 must-know/important/noise、事件边界和证据关系；
- 建立当前管线的覆盖率、重复率、来源集中度、摘要质量和耗时基线；
- 增加数据质量报告命令，但不改变线上排序。

退出条件：能够用同一数据集比较旧管线与新管线，而不是凭页面观感判断。

### Phase 1：分类轴和来源注册表（核心完成）

工作：

- 引入 `primaryBeat/scope/eventType/topics`；
- 为来源增加角色、覆盖、验证策略和健康字段；
- 建立 coverage matrix 和健康感知调度器；
- 保留旧 `primaryCategory` 兼容投影。

退出条件：每个 beat 的发现/验证入口清晰，Serverless 不再固定切前 6 个来源。

### Phase 2：候选标准化和事件模型（核心完成）

工作：

- 拆分抓取、标准化、质量门槛和来源归因；
- 引入 `StoryCluster`、`Evidence`、事实状态和持续更新；
- 用实体、时间和语义相似度替换单纯 token overlap；
- 增加聚类解释和人工 fixture。

退出条件：同一事件跨来源只生成一个事件，跨语言/改写标题的聚类准确率达到 golden dataset 门槛。

### Phase 3：重要性分级和集合级选择（核心完成）

工作：

- 实现 must-know/important/special-interest/noise 分级；
- 分离重要性、置信度和用户偏好；
- 加入 beat/source/entity 多样性约束；
- 在 shadow mode 中同时生成 V1/V2 报告。

退出条件：V2 在 must-know 召回率、首页价值精确率和重复率上达到建议阈值，并优于 V1。

### Phase 4：事实约束摘要和 V2 API/UI（核心完成）

工作：

- 生成“发生了什么/为什么重要/关键事实/关注点”卡片；
- 加入数字、人名、日期和出处一致性校验；
- 扩展 `DailyNewsReportV2`，保留 `items` 兼容字段；
- 首页改为四层信息结构，分类页引用事件而不是复制文章。

退出条件：摘要校验、中文化、API 兼容和浏览器体验全部通过。

### Phase 5：持久化、可靠性和灰度切换（部分完成）

工作：

- 实现本地和生产 `NewsStore`；
- 将抓取从请求路径移到定时任务；
- 加入 last-known-good、质量门槛、熔断和回滚；
- 先内部 shadow，再小比例启用 V2，最后切换默认。

退出条件：连续 7 天刷新稳定，旧报告可一键恢复，V1 兼容字段在观察期内无调用后再考虑移除。

## 测试策略

| 层级 | 必须验证 |
| --- | --- |
| 单元测试 | 来源角色、时间、域名、质量门槛、事件分级、覆盖约束、摘要事实校验 |
| Fixture 集成测试 | 跨来源聚类、跨语言标题、事件更新、冲突证据、低价值噪声、单源爆料 |
| Golden dataset | must-know 召回、首页精确率、重复率、来源集中度和分类覆盖 |
| API 契约测试 | V2 分层字段、V1 `items` 兼容、空状态、last-known-good 和安全字段 |
| 性能测试 | 49 来源任务并发、超时、熔断、30 秒预算、API 读取延迟 |
| 浏览器测试 | 今日必知、重要进展、持续关注、分类引用、来源与事实状态展示 |
| Shadow 对比 | 同一天 V1/V2 报告差异及人工评分，不影响当前用户 |

详细实施时同步更新 [test-plan.md](test-plan.md)。

## 风险和缓解

| 风险 | 缓解 |
| --- | --- |
| 追求覆盖导致低价值填充 | 只为合格候选提供 coverage floor；没有合格事件时允许为空并说明原因 |
| 重要性规则固化编辑偏见 | 分离事实置信度、公共重要性和用户偏好；保留解释字段和 golden review |
| 聚类过度合并不同事件 | 实体、时间、事件类型共同约束；保存 evidence 并提供 fixture 回归 |
| 聚类不足造成重复 | canonical URL、语义相似度、实体与时间窗口联合判断 |
| 模型摘要产生幻觉 | 事实白名单、字段一致性校验、抽取式 fallback、来源归因 |
| 单一来源控制叙事 | 来源角色和独立证据要求；来源/实体多样性约束 |
| 抓取延迟或被封 | 健康度调度、熔断、last-known-good、公开元数据优先、不绕过访问控制 |
| 持久化增加运维复杂度 | `NewsStore` 抽象、本地适配器、最小保留期、供应商决策单独确认 |
| V2 一次性替换导致回退 | V1/V2 shadow、兼容 `items`、feature flag 和可恢复旧报告 |

## 非目标

- 不复制新闻网站全文，也不绕过付费墙、登录或访客验证。
- 不承诺绝对完整或绝对客观；目标是可测量地覆盖绝大多数 must-know 事件。
- 不让社交媒体单点消息直接成为已确认的核心新闻。
- 不用语言或地区作为新闻价值和用户偏好维度；它们只用于覆盖和处理。
- 不让大模型独自完成事实核验、来源可信度或 must-know 决策。
- 本轮不引入外部数据库供应商、用户账号、历史日报 API、生产调度器或监控平台。

## 验收标准

本地核心实现进入生产运营验证，需要满足：

- 用户认可“一站式”的可量化定义和建议内容层级；
- 确认栏目/beat 方案和每日大致内容量；
- 确认是否接受持久存储和异步定时采集；
- 确认是否允许模型参与事实约束的翻译/摘要；
- Phase 0 的 golden dataset 和 V1/V2 shadow 机制先于线上切换；
- 所有阶段都有可执行退出条件和 V1 回滚路径。

## 待确认

| 项 | 推荐默认 | 影响 |
| --- | --- | --- |
| 首页事件数量 | 今日必知 8–12、重要进展 20–35、持续关注 5–10 | 信息密度和阅读时长 |
| 最终 beat | 使用主题轴，地理拆成 `scope`；AI 暂保留独立 beat | 分类导航和来源矩阵 |
